The nudge carries the review. The verifier rubber-stamps it.

Q1 · The nudge feature

Not a noise source — the main yield engine. Its cost is wasted compute.

Nudges produce 75% of all findings, because the base pass finds nothing in 60% of reviewer lanes. Nudge findings are no less valid than base findings. But 68% of nudge work just re-finds known issues, and round 3 is nearly dead.

Q2 · The verifier agent

A good skeptic, a lenient confirmer — and overconfident.

Refutations are right 88% of the time. But it confirms 73% of everything at 0.93 confidence, and 1 in 3 confirmations is false. Same finding, same code, flips verdict on 38–48% of re-runs.

Method: logs parsed for per-round finding counts and per-finding verdicts; base/nudge provenance recovered from verify index order (engine appends nudge findings after base). Both MR diffs extracted from the logs; recurring findings re-judged blind (verdict hidden) by 2–3 adversarial Opus judges reading the actual code. Two judge workflows, ~150 agents, ~4.2M tokens. “Valid / actionable” = judge ground truth, not the verifier's own gate.

The headline “75% of findings come from nudges” sounds alarming until you see why: the first reviewer pass usually whiffs. In 6 lanes per run, the base pass returns nothing 60% of the time. The nudge rounds aren't piling noise onto a productive base — they're doing the discovery. (Partly a timing artifact of the reasoning-extract / “Compiling Findings to Nudge” flow: the base reasoning may already hold the issue that only gets compiled during a nudge.)

Q1 verdict. Nudge's “noise” is wasted compute (68% duplicate re-finding, a near-dead round 3), not output junk — its findings are as valid as the base pass's. The cheap win: set nudge_count: 3 → 2. Question 02

How well does the verifier actually verify or refute?

What it does: confirms almost everything, with high confidence

Confirmed — kept Refuted — dropped (policy: refuted-only ≥0.7) Unverified — kept
72.6% 21.4% 6%

Mean confidence: confirmed 0.93 · refuted 0.92 · unverified 0.72. Only refutations drop findings, so ~21% of findings are removed at this gate; everything else survives. (Minor bug: 13 verdicts emitted confidence on a 0–100 scale instead of 0–1.)

It is non-deterministic on identical input

Because the same two MRs were reviewed 80× and 57×, we can re-verify the same finding many times. It flips verdict on 38% (kopieerapparaat) / 48% (omvormer) of recurring issues. The flagship:

“File templates use double quotes instead of shellQuote” verified 53×
32 confirmed 12 ref. 9 unv.

Byte-identical input, three different verdicts. High confidence throughout. This is overconfidence, not nuance.

Ground truth: precise when it refutes, sloppy when it confirms

Blind judges read the real diffs and ruled each finding actionable or not, without seeing the verifier's verdict:

66%
Confirmation precision. 29 / 44 confirmed findings are actually actionable → 34% false-confirms
88%
Refutation correctness. 23 / 26 refuted findings really are non-issues; the 3 misses were all borderline nits

The failure is asymmetric. The verifier is a competent skeptic but a credulous confirmer — and since the drop policy only acts on refutations (which are rare), the false-confirms sail straight through to output. The tell: in flip cases, the verifier's minority refutation was often the correct verdict, outvoted by confident-but-wrong confirmations.

False confirmations — what slips through

humanReadableBytes panics on values ≥ 1 EiB confirmed 5/6not real

Impossible. "KMGTPE" is indexed by exp; reaching index 6 needs b ≥ 2⁷⁰, but int64 maxes at 2⁶³−1, so exp ≤ 5 always. No input can panic. The lone refuter was right.

executeResticCMDTemplate swallows template execution errors confirmed 69×not real

Hallucinated code. The function body isn't in the diff at all — only its call sites, which correctly check the returned error. The verifier confirmed (69 times, across 5 reworded variants) a defect in code it never saw.

Cleanup command uses unquoted $TMP variable confirmed 11×not real

Inverted reading. The diff fixes the quoting — every added line uses "${TMP}". The unquoted form exists only on the deleted lines. The finding flags the problem the change removed.

Registry endpoints lack authentication and authorization confirmed 5/6not real

Location-less generic claim. No HTTP endpoints are added in this diff; it actually adds correctly-scoped Kubernetes RBAC. A plausible-sounding security boilerplate with nothing behind it.

Correct refutations — what it rightly kills

List() is a breaking change to blobstore.Interface refutedcorrect

The sole implementer is updated in the same diff, it's an internal package with no external implementors, and the module builds clean. A normal interface evolution dressed as a bug — correctly dismissed.

fmt.Fprintf error handling on bytes.Buffer is overly strict refutedcorrect

There is no fmt.Fprintf anywhere in the diff. The verifier correctly refused a finding about a code path that doesn't exist.

Q2 verdict. The verifier is a reliable filter against hallucinations and non-issues (88% refute-correct) but a weak quality gate (66% confirm-precision, overconfident, non-deterministic). Because it confirms liberally and the drop policy only removes refutations, roughly 1 in 3 surviving findings is a false-confirm. It's catching the wrong errors with the wrong confidence.

So what

Highest-leverage changes

  • 1
    Cut nudge_count 3 → 2. Round 3 yields 12% of new findings for a full extra reviewer pass per lane, empty 77% of the time. Near-free compute saving with negligible recall loss.
  • 2
    Make the confirm gate as adversarial as the refute gate. Most false-confirms are mechanically catchable: force “is this code even in the diff?”, “trace the type / index bound”, “is the finding describing the deleted or the added side?” before allowing confirmed.
  • 3
    Verify each finding 2–3× and require agreement. The verifier already disagrees with itself 38–48% of the time; majority-vote turns that noise into a signal and would have killed the EiB / registry-auth confirmations.
  • 4
    Stop trusting the confidence number. 0.93 on confirmations that are 66% correct means confidence is uncalibrated. Either recalibrate, or extend the drop policy toward refuted-and-unverified and lean harder on the verdict confidence threshold.
  • 5
    Separately: raise the finding bar. The reviewer prompt is producing mostly “add a test” / style notes (only 3 of 29 valid findings are medium+). That's a reviewer-prompt/priority issue, independent of nudge and verifier.
